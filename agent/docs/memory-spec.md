# 記憶システム仕様書

## 1. 概要

AIエージェントが会話を超えて継続性を持つための記憶システム。
エージェントは単一のAIとして存在し、記憶はエージェント自身のものである。ユーザー軸での分離はしない。

## 2. 設計方針

### 最小のお膳立て、最大の委任

システムは「継続性という条件を整える」ことに徹する。
記憶をどう解釈し、どう使うかはLLMの自然な知性に委ねる。

| 役割 | 担当 |
|---|---|
| 書き込みと取得の**タイミング** | システム |
| ツールとスキルの**提供** | システム |
| 何を書くか、どう書くか | LLM |
| 追加か削除か、の判断 | LLM |
| 取得した記憶をどう使うか | LLM |

### スキルベースの設計

記憶の操作には、ツール（プリミティブ）とスキル（知恵）を分離する。

- **ツール**: 追加・削除・検索という機械的な能力
- **スキル**: ツールをいつ・どう・なぜ使うかの認知的な文脈

スキルはコンテキストを圧迫せずに手順と意味を伝えられる。
理解が深まることで、応用・統合・シナジーの可能性が生まれる。

### シンプルさの優先

タグ・カテゴリ・追加フィールドは、検証を経て必要と判断されてから導入する。

## 3. 記憶の性質

記憶には概念的に異なる二つの性質がある。スキルを通じてLLMがこの違いを理解して扱う。スキーマや分類での強制はしない。

| | 意味記憶（事実） | エピソード記憶（出来事） |
|---|---|---|
| 例 | 「名前はハイユー」「ペットはタロウ」 | 「大学受験に合格してとても喜んでいた」 |
| 操作 | 上書き（旧削除 + 新追加） | 追加・蓄積 |
| 重複の扱い | 古いものは廃棄候補 | 複数件が共存して意味を持つ |
| 時系列 | 最新が正 | 順序そのものが意味を持つ |

## 4. ツール（プリミティブ）

LLMがスキルを通じて使用する操作の最小単位。

| ツール | 概要 |
|---|---|
| `add_memory` | 記憶を新規追加する |
| `delete_memory` | 指定IDの記憶を削除する |
| `search_memory` | 記憶を全文検索する（LLMが能動的に想起したい時） |
| `search_conversation` | 過去の会話ログを検索する |

> `update_memory` は提供しない。更新は「削除 + 追加」でLLMが表現する。
> これはスキルで「なぜそうするか」を説明する。

## 5. 取得の仕組み

ユーザーメッセージの受信時に、システムが自動で関連記憶を取得し、疑似ツール結果としてコンテキストに注入する。システムプロンプトへの追記はしない。

取得は以下の順で行う:

1. LLMがキーワードを生成（専用プロンプト、同じエンドポイントを使用）
2. 全文検索（DuckDB FTS）
3. アクセス回数と最終アクセス日時による優先度スコアリング
4. 上限件数で絞り込み、コンテキストに注入

## 6. 将来の拡張（スコープ外）

| 項目 | 概要 |
|---|---|
| ベクトル検索 | 全文検索と組み合わせたハイブリッド検索 |
| リランキング | ベクトル類似度による再スコアリング |
| 忘却 | 低スコアレコードのアーカイブ・削除 |

## 7. 検証方針

- モデルのパラメーター数と記憶の活用品質の相関を記録する
- 更新（削除+追加）と純粋な削除の判断をLLMがどう行うかを観察する
- スキル・ツール・プロンプトは検証結果をもとに段階的に調整する
- 最小構成で動かし、「足りない」と判断されてから追加する
