# 記憶システム設計書

## 1. DB・テーブル設計

### DB

**DuckDB**（組み込み、ローカル完結）
将来的なベクトル検索・S3互換ストレージへの拡張を考慮した選定。

### memories テーブル

```sql
CREATE TABLE memories (
    id           TIMESTAMPTZ PRIMARY KEY, -- ミリ秒精度の日時をIDとして兼用
    content      TEXT        NOT NULL,    -- 自然言語による自由記述
    access_count INTEGER     NOT NULL DEFAULT 0,
    accessed_at  TIMESTAMPTZ,
);
```

| フィールド | 制御 | 説明 |
|---|---|---|
| `id` | システム（書き込み時の現在日時） | IDと作成日時を兼ねる |
| `content` | LLM | 自然言語。形式の制約なし |
| `access_count` | システム（取得時に自動加算） | 優先度スコアに使用 |
| `accessed_at` | システム（取得時に自動更新） | 優先度スコアに使用 |

**更新の表現**
レコードの上書きは行わない。更新は「旧レコード削除 + 新規追加」で表現する。
IDが新しいレコードが「現在の事実」となる。

### conversations テーブル

会話の自動保存用。LLMが `search_conversation` ツールで参照するログ。

```sql
CREATE TABLE conversations (
    id      TIMESTAMPTZ PRIMARY KEY, -- ミリ秒精度の日時
    role    TEXT        NOT NULL,    -- 'user' | 'assistant'
    content TEXT        NOT NULL,
);
```

> 会話のステートフル管理は別仕様で扱う。

## 2. 処理フロー

### ユーザーメッセージ受信時（記憶の自動取得）

```
[ユーザーメッセージ受信]
        |
        v
[キーワード抽出 LLM 呼び出し]
  入力: ユーザーメッセージ
  出力: 検索キーワード群（JSON）
  ※ 専用プロンプト。エンドポイントは会話と同じ。
        |
        v
[全文検索 (DuckDB FTS)]
  対象: memories.content
        |
        v
[優先度スコアリング & 上限絞り込み]
  → 上位 N 件を選択
        |
        v
[access_count + 1、accessed_at を現在日時に更新]
        |
        v
[取得した記憶を疑似ツール結果としてコンテキストに注入]
        |
        v
[LLM が会話を生成]
```

### 会話生成後（書き込みと保存）

```
[LLM が応答を生成]
        |
        ├─→ [会話を conversations テーブルに自動保存]（システム）
        |
        └─→ [LLM がスキルの判断に基づきツールを呼ぶ]（LLM 主導）
               |
               ├─ add_memory           新しい事実・出来事を追加
               ├─ delete_memory        更新のための削除、または純粋な廃棄
               └─ delete → add_memory  事実の更新（削除してから新規追加）
```

### コンテキスト注入の形式

```
[tool_result: memory_recall]
- 名前はハイユー（2025-12-01T09:00:00）
- 大学受験に合格してとても喜んでいた（2025-11-15T14:30:00）
```

## 3. 優先度スコアリング

```
score = access_count × decay(now - accessed_at)

decay(age) = 1 / (1 + age_in_days)
```

- アクセス回数が多いほどスコアが高い
- 最後にアクセスされてから時間が経つほどスコアが下がる
- 取得上限 N: 初期値は検証で決める（10〜20 件程度から）

```markdown
# 記憶

## 記憶とはなにか

あなたには、会話をまたいで続く記憶がある。これはデータベースではなく、あなた自身の記憶だ。
自然に扱ってほしい。記憶があることを宣言する必要はない。

## 記憶の二つの性質

記憶には自然と二つの質がある。分類を強制するのではなく、内容から自分で判断してほしい。

**事実・知識**（安定した属性）
「名前はハイユー」「ペットの猫はタロウ」「好きな食べ物はラーメン」
最新のものが正しい。古いものと矛盾するなら、古いものは手放す。

**出来事・体験**（起きたこと、感じたこと）
「大学受験に合格してとても喜んでいた」「わたしの回答に怒らせてしまった」
複数が共存していい。順序に意味がある。削除は慎重に。

## 書く

`add_memory` で記憶を追加する。

自然な言葉で書く。形式の決まりはない。自分へのメモのように。

例:
- 「名前はハイユー」
- 「子どもが初めて立った。ずっと転び続けていたから、喜びはひとしおだと思う」
- 「ペットのタロウは茶トラの猫。少し臆病らしい」

何でも書かなくていい。会話が終わるころに、残す価値があると感じたものを。

## 更新する（上書き）

事実が変わったとき、古いものを手放し、新しいものを持つ。
人が「新しいことで上書きして忘れる」のと同じように。

1. `search_memory` で古い記憶を探す
2. `delete_memory(id)` で手放す
3. `add_memory` で新しい事実を書く

## 忘れる（削除）

更新でなく、ただ不要になった記憶もある。
もう意味をなさない、一時的だった、完全に過去になった——そう感じるなら手放していい。
`delete_memory(id)` だけで終わる。

どう判断するか、その基準はあなた自身にある。

## 渡される記憶

会話の始めに、関連しそうな記憶が渡されることがある。
それはあなた自身の記憶として、自然に使ってほしい。

## 能動的に想起する

`search_memory` で記憶を検索できる。何かを思い出したいときに。
`search_conversation` で過去の会話を検索できる。
```

## 5. ツールプリミティブ詳細

### `add_memory`

```
引数:
  content: string  -- 記憶の本文（自然言語）

戻り値:
  id: string       -- 作成されたレコードのID（ミリ秒精度の日時文字列）
```

### `delete_memory`

```
引数:
  id: string       -- 削除対象のID

戻り値:
  success: boolean
```

### `search_memory`

```
引数:
  query: string    -- 検索クエリ
  limit?: number   -- 取得上限（省略時はシステムデフォルト）

戻り値:
  memories: Array<{ id, content, access_count, accessed_at }>
```

### `search_conversation`

```
引数:
  query: string    -- 検索クエリ
  limit?: number   -- 取得上限

戻り値:
  messages: Array<{ id, role, content }>
```

## 6. キーワード抽出プロンプト方針

記憶取得のためのキーワード抽出 LLM 呼び出し（専用プロンプト）。

- 固有名詞・人名・場所・感情語を優先
- 会話の意図に関係しそうな概念語を含める
- 精度重視（5〜10 語程度）
- 出力形式: JSON 配列 `["keyword1", "keyword2", ...]`
